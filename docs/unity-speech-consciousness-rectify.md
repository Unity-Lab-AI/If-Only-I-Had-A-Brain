#### SPEAK.1 — ⛔ ROOT CAUSE of "word salad after 9th grade": word_motor bucket-band DRIFT as vocabulary grows

**Defect:** `js/brain/cluster/emit.js:598` — `bucketSize = max(1, floor(subjSize / wordsList.length))` is recomputed on EVERY emit from the CURRENT word-list length. `_ensureWordBucketMap` (`js/brain/curriculum.js:10952-10984`) is correctly append-only ("Existing bucket positions never renumber — which would invalidate trained sem_to_word_motor weights") — but that comment's guarantee is FALSE at read time: the bucket INDEX is stable, the PHYSICAL NEURON BAND `bStart = subjStart + b*bucketSize` (`emit.js:611`) is not. Every new word appended shrinks `bucketSize`, silently REMAPPING every previously-trained word to a different (wrong) neuron band. Teach-side writes (`_writeAnswerToWordMotor` `curriculum.js:11004-11012` uses the same live division) drift in lockstep with emit, so training keeps "passing" while ALL prior grades' sem→word_motor bindings point at stale bands. Small vocab (K-G3) barely moves the division → she speaks fine; by grade 9 the accumulated dictionary (per-grade vocab files: `js/brain/grade9-vocabulary.js` etc., 45.8k total) has re-divided the band dozens of times → the argmax reads mostly desynced weights → topically-nearby but sequence-random output = word salad. Secondary capacity collapse: thousands of words over a fixed sub-band → `bucketSize→1` cell per word (argmax over 1 noisy cell) and word indices past `subjEnd` become physically unreadable (`emit.js:612` clamp).

**Rectify (equational, no templates):**
- **SPEAK.1a — freeze bucket geometry at first allocation.** Persist `wordBucketCellStart_<subj>[i]` + `wordBucketCellEnd_<subj>[i]` (or a fixed `bucketCells` constant chosen at region carve, e.g. 8 cells/word) when a word is FIRST assigned. Emit + teach + `_writeAnswerToWordMotor` all read the frozen geometry. New words append into the NEXT free band; never re-divide occupied bands.
- **SPEAK.1b — capacity ledger + region growth.** When the sub-band's free cells run out, grow `word_motor_<subj>` at the region carve (auto-scale already sizes regions) OR spill into an overflow band — never shrink existing buckets. Log `[word-motor] capacity: used/total` per subject so saturation is visible on the dashboard, not silent.
- **SPEAK.1c — migration for already-trained weights.** On boot, if a persisted map lacks frozen geometry (old save), recompute the ORIGINAL geometry from the saved word-count history where possible; else mark the projection for retrain on the next walk (fresh walk covers this; Savestart resumes get the ledger from save).
- **SPEAK.1d — persist the bucket maps + geometry in saveWeights/loadWeights** (verify `wordBucketWords_<subj>` and the new geometry arrays survive restart; a resumed brain with a re-enumerated dictionary order is the same drift bug in disguise — `_enumerateBucketableWords` `curriculum.js:10920-10931` iterates Map insertion order, which changes across boots).

**✅ SHIPPED (code, 2026-07-01, branch `feature/unity-speech-consciousness-rectify`):** single authority `cluster.wordBucketCellSizeFor(subj)` (`js/brain/cluster/emit.js`) freezes cells-per-word = `bandSize / DREAM_WORD_MOTOR_VOCAB_CAP` (default 50000) once per subject, deterministic across boots.
